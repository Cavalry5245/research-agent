## 为什么需要性能测试

在大厂面试中，面试官会问：

- **"你的系统能支撑多少QPS？"** - 没有测试数据 = 不专业
- **"P95延迟是多少？"** - 不知道 = 缺乏生产经验
- **"如何定位性能瓶颈？"** - 没有工具 = 空谈架构

性能测试让你能用**数据说话**，而不是"应该挺快的吧"。

## 性能测试策略

### 三层测试金字塔

```
         ┌──────────────┐
         │  压力测试     │  ← 找极限（每周）
         │  Stress Test  │
         └──────────────┘
       ┌─────────────────────┐
       │  端到端测试           │  ← 真实场景（每次发版）
       │  E2E Performance     │
       └─────────────────────┘
   ┌──────────────────────────────┐
   │  组件性能测试                 │  ← 快速反馈（每次PR）
   │  Unit Performance Tests       │
   └──────────────────────────────┘
```

### 核心指标（面试必问）

| 指标 | 解释 | 面试说法 |
|------|------|---------|
| **P50** | 50%的请求延迟 | "中位数延迟500ms" |
| **P95** | 95%的请求延迟 | "95%的用户在1秒内得到响应" |
| **P99** | 99%的请求延迟 | "长尾延迟2秒" |
| **QPS** | 每秒查询数 | "单机支持10 QPS" |
| **Throughput** | 吞吐量 | "每秒处理100个chunks" |

## 测试套件说明

### 1. `test_embedding_perf.py` - Embedding性能

**测什么**：
- 单次查询延迟（不同文本长度）
- 批量吞吐量
- 冷启动 vs 热启动

**关键指标**：
- P95延迟 < 100ms
- 吞吐量 > 5 texts/sec

**面试话术**：
> "我测试了不同文本长度的embedding延迟，发现短文本（~20 tokens）P95在50ms，长文本（~500 tokens）P95在120ms。冷启动比热启动慢2.5倍，所以我加了连接预热。"

### 2. `test_vector_store_perf.py` - 向量存储性能

**测什么**：
- 索引速度（不同数据规模）
- 查询延迟（数据库大小影响）
- 并发查询性能
- 内存占用

**关键指标**：
- 5000 chunks查询P95 < 500ms
- 内存增长 < 100MB/1000 chunks

**面试话术**：
> "我做了可扩展性测试，发现Chroma在5000 chunks时查询延迟300ms，但到10000时涨到800ms。如果数据量继续增长，需要考虑Milvus等分布式方案。"

### 3. `test_qa_e2e_perf.py` - QA端到端性能

**测什么**：
- 延迟分解（embedding + 检索 + LLM）
- 延迟分布（P50/P95/P99）
- QPS吞吐量
- 并发性能
- 缓存效果

**关键指标**：
- P95延迟 < 2000ms
- QPS > 1

**面试话术**：
> "我用延迟分解法发现LLM生成占73%，是主要瓶颈。于是我优化了prompt长度，从2000 tokens降到800，P95延迟从1.8秒降到0.9秒。"

### 4. `test_stress.py` - 压力测试

**测什么**：
- 最大并发数
- 持续负载稳定性
- 内存泄漏检测
- 错误恢复能力

**关键指标**：
- 找到最佳并发数（QPS最高点）
- 100次查询内存增长 < 100MB

**面试话术**：
> "我做了压力测试，发现最佳并发数是5，此时QPS达到峰值8.5。超过5个worker后由于LLM API限流，QPS反而下降。同时验证了100次查询后无内存泄漏。"

## 如何解读结果

### 延迟分解示例

```
=== QA Latency Breakdown ===
Embedding:    50.23ms (10.2%)
Retrieval:    80.45ms (16.3%)
LLM Gen:     362.18ms (73.5%)
Total:       492.86ms
```

**分析步骤**：
1. **找瓶颈**：LLM占73.5%，是主要瓶颈
2. **评估改进空间**：即使embedding和检索优化到0，总延迟只能降到362ms
3. **优化方向**：
   - 短期：减少LLM生成token数
   - 中期：使用更快的模型
   - 长期：添加结果缓存

### 并发测试示例

```
=== Concurrent Query Performance ===
 1 workers: QPS=1.20
 2 workers: QPS=2.10
 5 workers: QPS=4.50  ← 最佳
10 workers: QPS=3.80  ← 开始下降
20 workers: QPS=2.50
```

**分析**：
- 5 workers是最佳点（QPS=4.50）
- 超过5后下降 → LLM API成为瓶颈（可能有限流）
- **生产配置**：设置并发池=5

## 优化建议

### 场景1：延迟高（P95 > 2秒）

**诊断**：
```bash
# 运行延迟分解测试
pytest tests/performance/test_qa_e2e_perf.py::TestQAEndToEndPerformance::test_qa_latency_breakdown -v -s
```

**优化方向**：
1. **LLM是瓶颈** → 减少prompt、用更快的模型、加缓存
2. **Embedding是瓶颈** → 用本地模型、GPU加速、批量处理
3. **检索是瓶颈** → 优化索引、减少top_k、用Milvus

### 场景2：QPS低（<1）

**诊断**：
```bash
# 运行吞吐量测试
pytest tests/performance/test_qa_e2e_perf.py::TestQAEndToEndPerformance::test_qa_throughput -v -s
```

**优化方向**：
1. 使用异步LLM调用（async/await）
2. 批量embedding
3. 连接池复用
4. 添加Redis缓存

### 场景3：内存占用高

**诊断**：
```bash
# 运行内存测试
pytest tests/performance/test_vector_store_perf.py::TestVectorStorePerformance::test_memory_usage -v -s
```

**优化方向**：
1. 向量量化（float32 → int8）
2. 分页加载
3. 定期清理缓存

## 面试准备

### 必备话术模板

#### 1. 介绍性能测试体系

> "我建立了三层性能测试：
> 
> - **组件测试**：测试embedding、向量存储的单元性能，每次PR都跑，反馈快
> - **端到端测试**：测试完整QA流程，发版前必跑，确保用户体验
> - **压力测试**：每周运行一次，找系统极限，检测内存泄漏
> 
> 所有测试结果自动生成报告，可以看历史趋势。"

#### 2. 展示问题解决能力

> "举个例子，刚开始QA P95延迟是2.3秒，不满足<2秒的目标。
> 
> 我做了延迟分解，发现：
> - Embedding: 50ms (2%)
> - 检索: 100ms (4%)
> - LLM: 2150ms (94%)  ← 瓶颈
> 
> 于是我做了三个优化：
> 1. Prompt工程：从2000 tokens压缩到800
> 2. 模型切换：从qwen-plus换到qwen-turbo
> 3. 结果缓存：相同问题直接返回
> 
> 最终P95降到820ms，QPS从0.8提升到2.1。"

#### 3. 展示系统思维

> "性能不是一次性工作，我建立了持续监控：
> 
> - **CI集成**：每个PR都跑quick_check，防止性能回退
> - **基准对比**：保留历史数据，可以看性能趋势
> - **生产监控**：部署Prometheus，P95延迟>2s自动告警
> 
> 这样既能保证开发时不引入性能问题，又能在生产及时发现问题。"

### 常见追问及答案

**Q: 如何确定性能目标？**
> "我参考了行业benchmark和用户体验研究：
> - <1秒：用户无感知等待
> - 1-3秒：可接受
> - >3秒：明显卡顿
> 
> 所以我设定P95<2秒，留buffer。同时QPS目标基于预估流量，单机1 QPS支撑100个活跃用户够用。"

**Q: 遇到性能瓶颈怎么办？**
> "分三步：
> 1. **定位**：延迟分解找到是哪一层的问题
> 2. **验证**：单独测试那一层，确认是真瓶颈
> 3. **优化**：先低成本优化（算法、缓存），再考虑硬件（GPU、分布式）"

**Q: 如何保证生产环境性能？**
> "三个层面：
> 1. **测试环境尽量接近生产**：用真实数据规模测试
> 2. **灰度发布**：先给5%流量，监控指标正常再全量
> 3. **快速回滚**：发现P95突增立即回滚，保留上个版本镜像"
